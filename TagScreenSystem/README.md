# TAG & SCREEN SYSTEM

## ОПИСАНИЕ
Система управления тегами и экранами для мониторинга данных в реальном времени с возможностью просмотра графиков и таблиц. Разработана на C# для работы в консоли Windows.

---

## ТЕХНИЧЕСКИЕ ХАРАКТЕРИСТИКИ

**TAG SERVICE (СЕРВИС ТЕГОВ)**
- Всего тегов: 2000
- Быстрые теги: 30 шт (обновление каждые 17 мс)
- Обычные теги: 1970 шт (обновление каждые 50 мс)
- Хранилище: JSON файл (Database/tags.txt)
- Автосохранение: каждые 5 секунд
- Queue Size: 1-20 (порог обновлений для прогресс-бара)

**SCREEN SERVICE (СЕРВИС ЭКРАНОВ)**
- Всего экранов: 12
- Графики: 5 шт (экраны 0-4)
- Таблицы: 7 шт (экраны 5-11)
- Тегов на экране: 300-500 (случайный выбор)
- Автообновление: каждые 250 мс

**ОТОБРАЖЕНИЕ ГРАФИКОВ**
- Формат вывода: [X; Y] где X = время, Y = значение
- Диапазон Y: [-2; 0]
- Каждая точка графика выводится на отдельной строке

**ПОЛЬЗОВАТЕЛИ**
- Admin: полный доступ, может редактировать значения тегов
- Operator: только просмотр экранов и графиков

## УСТАНОВКА И ЗАПУСК

**Требования:** .NET 6.0 или выше

**Пошаговая инструкция:**

1. Создайте папку проекта:
   mkdir TagScreenSystem
   cd TagScreenSystem

2. Создайте структуру папок:
   mkdir Models
   mkdir Database

3. Создайте файл проекта:
   dotnet new console -n TagScreenSystem -f net6.0

4. Скопируйте файлы кода в проект:
   - Program.cs
   - TagService.cs
   - ScreenService.cs
   - Models/Tag.cs
   - Models/Screen.cs

5. Скомпилируйте проект:
   dotnet build

6. Запустите приложение:
   dotnet run

## ВОПРОСЫ И ОТВЕТЫ

**1. Какой алгоритм используется в коде с тегами?**

Используется асинхронный многопоточный алгоритм с динамической корректировкой задержки. Два параллельных потока обновляют теги: один 30 быстрых тегов с интервалом 17 мс, второй 1970 обычных тегов с интервалом 50 мс. Время выполнения кода вычитается из целевой задержки для точного соблюдения интервалов. Теги хранятся в ConcurrentDictionary для потокобезопасности.

**2. Как рассчитывается значение Y на графике?**

Y рассчитывается по формуле: mappedValue = -2 + (currentValue % 100) / 100 * 2. Исходное значение тега от 0 до 100 преобразуется в диапазон от -2 до 0. Пример: значение 0 даёт -2, 50 даёт -1, 100 даёт 0.

**3. Что такое Queue Size и как он работает?**

Queue Size (1-20) - это порог обновлений, который нужно достичь для получения галочки ✓. Каждый тег имеет счётчик UpdateCounter, увеличивающийся при каждом обновлении. Каждые 250 мс счётчики обнуляются. Прогресс = min(UpdateCounter, QueueSize) / QueueSize * 100%. Галочка появляется когда UpdateCounter >= QueueSize.

**4. Почему 30 тегов обновляются каждые 17 мс, а 1970 каждые 50 мс?**

Это требование из технического задания. 30 быстрых тегов симулируют критически важные датчики (частота ~58 Гц). Остальные 1970 тегов - обычные датчики (частота 20 Гц).

**5. Как работает поиск по дате на графике?**

Алгоритм линейного поиска: перебираются все точки графика, для каждой вычисляется разница во времени с искомой датой, выбирается точка с минимальной разницей. Сложность O(n). Выводятся найденная точка и 3 соседние точки слева и справа.

**6. Как обеспечивается точность обновления 17 мс и 50 мс?**

Благодаря динамической корректировке задержки: замеряется время начала цикла, после обновления вычисляется затраченное время, затем задержка рассчитывается как targetInterval - elapsed. Если обновление заняло 3 мс, при интервале 17 мс программа подождёт 14 мс.

**7. Где можно использовать эту систему?**

Промышленная автоматизация и SCADA-системы: мониторинг датчиков на производстве (температура, давление, скорость), диспетчерские пункты, контроль качества, обучение операторов, анализ исторических данных.

**8. Как работает прогресс-бар в таблице?**

Прогресс-бар визуализирует сколько обновлений сделал тег за последние 250 мс относительно Queue Size. Пустой бар (░░░░░░░░░░) - обновлений не было. Частично заполненный (████████░░) - идёт накопление. Полный бар с галочкой (██████████ 100% ✓) - порог достигнут.

**9. Что такое NumberOfDisplays?**

NumberOfDisplays (10-500) - количество тегов, отображаемых на одном экране. Позволяет не перегружать экран информацией, ускорять отрисовку и фокусироваться на актуальных данных.

**10. Как происходит сохранение в базу данных?**

Сохранение происходит в JSON формате в файл Database/tags.txt при генерации тегов, при изменении значения тега (Admin) и автоматически каждые 5 секунд в фоновом потоке. При запуске данные загружаются из файла, если он существует и не повреждён.

**11. В чём разница между графиком и таблицей?**

График (экраны 0-4) отображает точки [X; Y] где X=время, Y=значение. Каждая точка на новой строке. Показывает динамику изменения значений. Таблица (экраны 5-11) отображает текущие значения всех тегов в виде таблицы с ID, типом, значением, прогресс-баром. Обновляется в реальном времени.

**12. Можно ли добавить новые теги в реальном времени?**

В текущей версии можно только редактировать значения существующих тегов (для Admin). Для добавления новых тегов нужно изменить метод GenerateTags() в TagService.cs и перекомпилировать проект.

**13. Как работает переключение между Admin и Operator?**

Operator видит все пункты меню кроме "Редактировать тег". Admin имеет доступ ко всем пунктам и может изменять значения тегов через пункт 7. Переключение происходит через пункт меню 6.

**14. Почему 12 экранов, а не от 3 до 12 как в ТЗ?**

В текущей версии зафиксировано 12 экранов для стабильности. Можно вернуть случайное количество, заменив в ScreenService.cs строки: int screenCount = _rand.Next(3, 13); int chartCount = _rand.Next(2, 6);

**15. Как остановить программу?**

В главном меню нажать 0. В LIVE режиме сначала нажать Q для выхода, затем 0. Принудительно: Ctrl + C в консоли.

**16. Что происходит при ошибке чтения базы данных?**

При повреждении файла Database/tags.txt выводится сообщение об ошибке, файл удаляется, создаётся новая база данных с 2000 тегов, программа продолжает работу.

**17. Какой алгоритм поиска ближайшей точки по дате?**

Используется алгоритм линейного поиска с метрикой L1: closest = data.OrderBy(d => Math.Abs((d.Time - targetDate).TotalSeconds)).First(); Сложность O(n).

**18. Как работает автообновление экранов?**

Запускается фоновый поток при создании ScreenService. Цикл: await Task.Delay(250) -> UpdateAllScreens(). UpdateAllScreens() обновляет только экраны-графики. Для каждого графика пересчитываются точки на основе текущих значений тегов.

**19. Как симулируются значения тегов?**

Используется генератор случайных чисел: tag.Value = _rand.NextDouble() * 100. В реальной системе здесь был бы код чтения с реальных датчиков (PLC, Modbus, OPC UA).

**20. Можно ли изменить интервалы обновления?**

Да, в TagService.cs в методе GenerateTags() измените UpdateIntervalMs для быстрых и обычных тегов. MaxUpdates пересчитается автоматически: 250 / UpdateIntervalMs.
